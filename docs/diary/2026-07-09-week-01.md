# Week 1 — 2026-06-30 → 2026-07-09

First entry, covering roughly the first nine days: project scaffolding through
to a working product detail page (PDP) with a client-side cart in progress.

## Summary

Went from an empty repo to a pnpm monorepo with three apps (Next.js frontend,
Express backend, standalone Sanity Studio) and a Supabase/Postgres schema. The
big architectural move this week was adopting **Sanity as the source of truth
for product/catalogue content**, while keeping orders, customers, and capacity
in Postgres. The PDP is now fetching real product data from Sanity, and the
groundwork for the cart (Zustand store, drawer UI, toast feedback) is in place
but not yet fully wired.

## Decisions & justifications

### Product content lives in Sanity, not Postgres

The catalogue (cakes, categories, sizes, flavours, colours, images) now lives in
Sanity's Content Lake instead of the Postgres database.

- **Images are the main driver.** Sanity's asset pipeline + image CDN gives us
  on-the-fly resizing, cropping, hotspot/focal-point control, and automatic
  format negotiation (WebP/AVIF) via a URL builder. Doing this ourselves on top
  of Postgres + object storage would have been a lot of undifferentiated work.
- **Editor experience.** Sanity Studio is a proper CMS UI, so non-technical
  admins can add/edit cakes, upload images, and manage variants without touching
  the database or an admin panel we'd have to build.
- **Content modelling fit.** Cakes are content-shaped (rich descriptions via
  Portable Text, media, flexible variant lists) rather than strictly relational.

Postgres still owns the transactional/relational data — `order`, `order_item`,
`customer`, and the capacity tables — because that's where integrity,
constraints, and Supabase Auth relationships matter.

### Linking the two systems: `order_item.sanity_variant_id`

Each priced **size** in a product's `sizes[]` array carries Sanity's
auto-generated `_key`. That `_key` is what an order line references via
`order_item.sanity_variant_id`, and `order_item.unit_price` is a deliberate
price **snapshot** (not a foreign key) so historical orders stay correct even if
prices change later. Sizes are the only variant dimension that affects price;
flavour and colour do not.

### Standalone Studio, not embedded

The Studio is its own app (`apps/studio`) rather than being mounted
inside the Next.js app. Rationale: far faster dev/build (Vite vs. compiling
through Next), automatic Studio updates, and TypeGen watch mode. It keeps the
content model independent of the website.

### Colour picker via `sanity-plugin-color-input`

Swapped a hand-rolled hex-string field for the community colour-input plugin so
admins get a real swatch picker. Stored value is a structured `color` object
(`hex`, `rgba`, `hsl`, gradient fields). Noted that the plugin is young/single-
maintainer, but it's the one listed in Sanity's official plugin directory.

### State architecture: scoped Context vs. global store

Two distinct kinds of state, handled two different ways:

- **PDP selection state** (selected size, flavour, colour, date, notes) lives in
  a page-scoped `ProductPurchaseContext` built on `useReducer`. It's naturally
  scoped to a single product page's lifetime, and the reducer keeps the growing
  set of related actions (`SELECT_SIZE`, `SET_DATE`, …) readable. Passing a
  Server Component as `children` into the client provider keeps the `"use
  client"` boundary small — e.g. only the price number (`SelectedPrice`) and the
  variant buttons are client, not the whole content block.
- **Cart state** is global and cross-page, so it lives in a **Zustand** store
  (`useCartStore`) with the `persist` middleware (localStorage). Chose Zustand
  over more Context because the cart and its open/closed UI state are read from
  unrelated places (header badge, drawer, PDP) where Context would mean provider
  nesting and broad re-renders. Used `partialize` so only `cartItems`/`total`
  persist — `isCartOpen` resets to `false` on reload rather than popping the
  drawer open on a fresh visit.

### UI building blocks

- **shadcn (Base UI variant)** for primitives — installed Popover + Calendar
  (date picker), Drawer (cart), and Sonner (toasts). The date picker enforces a
  5-day minimum notice via `disabled={{ before: … }}`.
- **Sonner** chosen over an inline "added to cart" alert: no layout shift,
  auto-dismiss, reusable app-wide, and supports an action button (e.g. a
  "View cart" shortcut). Pinned to the light theme to match the cream design and
  dropped the `next-themes` dependency the generator pulled in.

## What was built

- pnpm monorepo: `apps/frontend`, `apps/backend`, `apps/studio`,
  `db/schema.sql`.
- Sanity Studio with `product` (cake) and `category` schemas; sizes, flavours,
  colours, images, Portable Text description. Schema deployed; TypeGen wired so
  generated types feed the frontend.
- Frontend ↔ Sanity: `next-sanity` client, image URL builder, env vars, CORS.
- PDP: server-rendered `ProductDisplay` fetches once and prop-drills fields into
  `ProductGallery` (thumbnails + main image), `ProductContent`, `SelectedPrice`,
  and `ProductVariants`.
- Cart foundations: `useCartStore` (add/remove/clear, open/close/toggle,
  persisted), a dumb `Cart` drawer component, and Sonner mounted in the layout.

## What's next

Immediate priorities to make the cart actually work end-to-end:

- **Wire up "Add to Cart".** Finish connecting the PDP's add handler to the
  store so a configured cake becomes a persisted cart line, and fire a Sonner
  confirmation toast (with a "View cart" action).
- **Open the cart.** Connect the `Cart` drawer's `open`/`onOpenChange` to the
  store's `isCartOpen`/`openCart`/`closeCart`, and add a cart trigger (icon +
  item count) in the header.
- **Client-side validation before adding.** Require that **size, flavour, and a
  delivery date** are all selected before an item can be added — disable the Add
  to Cart button (or surface an inline/toast message) until the selection is
  complete.
